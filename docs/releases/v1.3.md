# Agent Handoff 1.3

Agent Handoff 1.3 adds explicit human control over containerization decisions and strengthens testing guidance for GUI-based projects.

## Highlights

- Mandatory separate containerization question during both new-repository initialization and adoption into an existing repository.
- Agents cannot infer, create, relocate, consolidate, or migrate Docker and Docker Compose infrastructure before the user confirms the approach.
- Complete containerization protocol in `ai/CONTAINERIZATION.md`.
- Supported layouts for no repository-managed containerization, colocated files, centralized `docker/`, hybrid projects, modular monorepos, separate deployment repositories, and established custom layouts.
- Existing container layouts are preserved unless migration is explicitly approved.
- The hybrid layout may be recommended for many small and medium multi-service repositories, but it is never selected automatically.
- Compose path, override, environment, migration, verification, project-memory, and handoff requirements.
- Prohibition on brittle automated GUI tests based on absolute coordinates, screen position, pixel offsets, or incidental layout order.
- Position-dependent GUI checks are performed manually or as supervised exploratory checks with Codex.
- Automated GUI tests should use semantic roles, accessible names, labels, stable component identifiers, or dedicated test IDs.

## Compatibility

Standard 1.3 does not require existing projects to reorganize Docker or Compose files.

When Agent Handoff is added to an established repository, the current container layout remains unchanged until the user explicitly authorizes a migration.

Projects that do not use Docker or Docker Compose do not need to create container files or `ai/CONTAINERIZATION.md`.
